## A ideia: camadas de abstração que se embrulham

A rede é organizada em **camadas**, igual ao software: cada uma esconde a de
baixo. O modelo de referência é o **OSI**, com 7 camadas. Sua mensagem **desce** a
pilha no remetente — cada camada adiciona seu próprio embrulho (cabeçalho) —
viaja como sinal físico, e **sobe** a pilha no destino, sendo desembrulhada camada
a camada.<sup><a href="#r1">1</a></sup>

_⬇️ remetente: aplicação → … → bits no fio · ⬆️ destino: bits → … → aplicação_

## As 7 camadas, cada uma como uma pergunta

Não decore a ordem — siga a lógica: cada camada responde **uma pergunta**
necessária para a mensagem viajar e ser entendida. As duas destacadas
(`load balancer`) são as que você vai usar de verdade.

<OsiStack />

<Callout variant="info">
  🧠 **O presente que a nuvem te dá:** on-premises você cuidaria das camadas 1 e 2
  (cabos, switches). Na AWS, **1 e 2 somem para você** e você opera da **camada 3
  (VPC)** para cima. É a vantagem "parar de manter data centers" vista pela lente
  das camadas.
</Callout>

## O foco: camada 4 vs camada 7 (NLB vs ALB)

De todas as 7, só duas decidem o tipo de load balancer — porque a pergunta é uma
só: **quão fundo o balanceador abre o pacote?**

<Callout variant="info">
  📨 **Analogia do envelope.** A **camada 4** é o **endereço escrito no envelope**
  (IP + porta): dá para entregar _sem abrir_. A **camada 7** é **abrir o envelope
  e ler a carta** (o caminho da URL, o cabeçalho `Host`, o cookie) para então
  decidir a rota.
</Callout>

|                    | NLB — camada 4                                  | ALB — camada 7                                            |
| ------------------ | ----------------------------------------------- | --------------------------------------------------------- |
| **O que enxerga**  | IP + porta (não abre o envelope)                | Conteúdo HTTP: URL, host, headers, cookies                |
| **Pode rotear por…** | destino TCP/UDP                               | **caminho** (`/api` → grupo X), **host**, método…         |
| **Velocidade**     | ⚡ extrema — milhões de req/s, latência mínima  | um pouco mais de overhead (ele lê e interpreta)           |
| **Protocolo**      | TCP / UDP (qualquer coisa)                      | HTTP / HTTPS (web)                                        |
| **Use quando…**    | performance/latência extrema, TCP não-HTTP, IP estático | app web, roteamento por conteúdo, redirecionar/autenticar |

<Callout variant="ok">
  **A regra de decisão em uma frase:** precisa que o balanceador _leia o HTTP_
  para decidir (rotear `/api` e `/site` para grupos diferentes)? → **ALB (L7)**. Só
  precisa de velocidade bruta encaminhando pacotes? → **NLB (L4)**. Mais
  inteligência custa um pouco de overhead; menos inteligência rende performance
  máxima.
</Callout>

<Callout variant="warn">
  **Cuidado com a intuição "L7 é melhor que L4":** não é melhor, é _diferente_. O
  NLB é superior justamente onde você _não_ quer que ninguém abra o envelope —
  latência mínima e throughput gigante. Quem decide é o trabalho, não o número da
  camada. (Mesma lógica do RI vs Savings Plans: não existe "o melhor", existe "o
  certo para a carga".)
</Callout>

### OSI vs. o modelo real (TCP/IP)

O OSI é didático (7 camadas). A internet roda o **TCP/IP**, mais enxuto (~4
camadas). "Camada 4" e "camada 7" sobrevivem no dia a dia porque existem nos dois
modelos e batem com o que os balanceadores fazem.<sup><a href="#r2">2</a></sup>

## Conectando à sua missão

Quando alguém perguntar _"por que ALB e não NLB?"_, a resposta defensável não é
"porque é o padrão" — é: **"nosso tráfego é HTTP e precisamos rotear por caminho
de URL para microsserviços diferentes; isso exige um balanceador de camada 7"**. E
numa investigação de incidente, a camada vira um localizador: "o ping responde (L3
ok) mas a porta recusa (L4)". Falar nessa régua é decidir e diagnosticar de igual
pra igual com infra.
